# GitHub Actions: CI, автодеплой на Beget и обновление событий

Краткая инструкция по тому, как в репозитории устроены проверки, выкладка статики на Beget (**FTP** или **SSH + rsync**) и зачем нужна ежедневная пересборка для блока «Что впереди» / «Прошедшие» на галерее.

См. также:

- [Миграция Strapi на Beget и webhook в GitHub](./strapi-cloud-to-beget.md) — DNS, FTP-секреты, PHP-реле для `repository_dispatch`.
- [События: дата, сортировка и cron](./events-date-sort-and-scheduled-deploy.md) — поле `event_at`, таймзона, логика planned/past при сборке.

---

## 1. Два workflow

| Файл | Имя в GitHub Actions | Когда запускается |
|------|----------------------|-------------------|
| [`.github/workflows/deploy.yml`](../.github/workflows/deploy.yml) | **CI** | Push и pull request в ветку `main` |
| [`.github/workflows/rebuild-static.yml`](../.github/workflows/rebuild-static.yml) | **Rebuild static site** | Вручную, webhook Strapi (`repository_dispatch`), по расписанию (cron) |

### 1.1. CI (`deploy.yml`)

**Цель:** убедиться, что **Astro** (lint, Prettier, `astro check`, сборка) проходит без ошибок. Код Strapi в этом репозитории нет — админка собирается в [отдельном репозитории CMS](repo-split-strapi-amvera.md).

**Деплой на Beget** выполняется **только** при одновременных условиях:

- событие — **push** в `main` (не PR);
- job **Astro** (`check-frontend`) завершилась успешно;
- в репозитории включена выкладка: **`BEGET_DEPLOY=true`** (рекомендуемое имя) **или** устаревшее **`BEGET_FTP_DEPLOY=true`** (оба варианта работают одинаково).

Job **Deploy Astro to Beget** скачивает артефакт со сборки Astro и заливает содержимое `dist/` на сервер (**FTP** или **rsync по SSH** — см. §2).

Если ни **`BEGET_DEPLOY`**, ни **`BEGET_FTP_DEPLOY`** не равны `true`, CI только проверяет и собирает — **на хостинг ничего не выкладывается**.

### 1.2. Rebuild static site (`rebuild-static.yml`)

**Цель:** пересобрать только фронтенд (Astro SSG) и при необходимости выложить на Beget **без** push в `main` (отдельный pipeline от обычного CI).

**Триггеры:**

1. **Вручную:** GitHub → Actions → **Rebuild static site** → **Run workflow**.
2. **Webhook:** внешний вызов GitHub API `repository_dispatch` с типом `strapi_publish` (через PHP на Beget, Make.com и т.д.) — см. [strapi-cloud-to-beget.md §6](./strapi-cloud-to-beget.md).
3. **Расписание:** каждый день в **03:00 UTC** (`cron: '0 3 * * *'`). Нужен для того, чтобы после смены календарного дня на сайте обновилось деление событий на «план» и «архив» без новой записи в CMS (подробнее в §3).

При **`BEGET_DEPLOY=true`** или **`BEGET_FTP_DEPLOY=true`** после сборки выполняется тот же деплой на Beget, что и в CI (метод — §2).

## 2. Секреты и переменные GitHub

**FTP или SSH:** по умолчанию используется **FTP** (как раньше). Если в **Variables** задать **`BEGET_DEPLOY_METHOD`** = **`ssh`**, выкладка идёт через **`rsync` по SSH** — обычно проще сопровождать на тарифе с SSH (один раз настроить ключ и путь, без пароля FTP в каждом запуске).

### Где именно прописать (интерфейс GitHub)

1. Откройте **репозиторий** на GitHub (тот, где лежат `.github/workflows/…`).
2. Вкладка **Settings** (настройки репозитория).
3. В левом меню: **Secrets and variables** → **Actions**.

Дальше два разных места:

| Что нужно | Вкладка | Кнопка |
|-----------|---------|--------|
| Пароли, токены, URL API, FTP | **Secrets** | **New repository secret** |
| Флаги `BEGET_DEPLOY` / `BEGET_FTP_DEPLOY`, способ `BEGET_DEPLOY_METHOD` (не секреты) | **Variables** | **New repository variable** |

Прямая ссылка (подставьте владельца и имя репозитория):  
`https://github.com/<OWNER>/<REPO>/settings/secrets/actions`

- Секреты: на той же странице откройте вкладку **Secrets**, затем **Actions** (если GitHub показывает подразделы) — в современном UI обычно одна страница с переключателем **Secrets** / **Variables** над таблицей.
- **Repository secrets** — то, что нужно для этих workflow: они доступны job’ам в **Actions** этого репозитория. Отдельно в коде, в `astro-app/.env` или в файлах репозитория секреты **не** дублируют — только в GitHub.

**Важно:** `STRAPI_URL` / `STRAPI_TOKEN` для **локальной** разработки по-прежнему задаются в `astro-app/.env` (не коммитьте токен). В **Secrets** GitHub — те же по смыслу значения, но для сборки в CI.

### Секреты (Secrets)

Общие для любого способа деплоя:

| Имя | Назначение |
|-----|------------|
| `STRAPI_URL` | Публичный URL Strapi **без** завершающего слэша, например `https://admin.example.ru` |
| `STRAPI_TOKEN` | API-токен Strapi с правами чтения нужных типов контента |

Если выкладка **по FTP** (переменная `BEGET_DEPLOY_METHOD` не задана или равна `ftp`):

| Имя | Назначение |
|-----|------------|
| `BEGET_FTP_HOST` | Хост FTP из панели Beget |
| `BEGET_FTP_USER` | Логин FTP |
| `BEGET_FTP_PASSWORD` | Пароль FTP |
| `BEGET_FTP_REMOTE_DIR` | Каталог на сервере для заливки **содержимого** `dist` (часто что-то вроде `./public_html/` — путь к корню **основного** сайта в панели) |

Если выкладка **по SSH** (`BEGET_DEPLOY_METHOD` = `ssh`) — на тарифе с SSH часто удобнее, чем FTP: быстрее, меньше сбоев, нормальная синхронизация каталогов.

| Имя | Назначение |
|-----|------------|
| `BEGET_SSH_HOST` | Хост SSH, как в `ssh user@host` (на Beget часто `логин.beget.tech`) |
| `BEGET_SSH_USER` | Логин SSH (часто совпадает с логином аккаунта Beget) |
| `BEGET_SSH_KEY` | **Приватный** ключ в одном блоке (включая строки `BEGIN … KEY` и `END … KEY`). Публичная часть должна быть в `~/.ssh/authorized_keys` на стороне Beget для этого пользователя |
| `BEGET_SSH_REMOTE_DIR` | Абсолютный путь к **корню сайта** на сервере, куда кладутся файлы из `dist` (например `/home/u1234567/ваш-домен.ru/public_html/`). Должен заканчиваться на `/`, чтобы `rsync` заливал **содержимое** `dist/` в этот каталог |

**Как завести ключ для CI:** на своей машине (один раз):

```bash
ssh-keygen -t ed25519 -C "github-actions-beget" -f ./beget_deploy_ed25519 -N ""
```

Содержимое **`beget_deploy_ed25519.pub`** добавьте в файл **`~/.ssh/authorized_keys`** на Beget под тем пользователем, под которым заходите по SSH (через панель или вручную по SSH). Содержимое **`beget_deploy_ed25519`** (приватный файл) целиком вставьте в секрет **`BEGET_SSH_KEY`**. Локальные файлы с ключом в репозиторий не коммитьте.

Проверка пути: зайдите по SSH и выполните `pwd` в каталоге сайта или посмотрите путь в файловом менеджере Beget к `public_html` основного домена — его и укажите в **`BEGET_SSH_REMOTE_DIR`**.

Без корректных `STRAPI_URL` и `STRAPI_TOKEN` сборка Astro в CI упадёт или соберётся с пустым контентом.

### Переменные (Variables)

| Имя | Значение | Эффект |
|-----|----------|--------|
| BEGET_DEPLOY` | `true` | Включает выкладку на Beget после сборки (**рекомендуемое** имя). |
| `BEGET_FTP_DEPLOY` | `true` | То же, что `BEGET_DEPLOY` (оставлено для совместимости со старыми инструкциями). Достаточно **одной** из двух переменных со значением `true`. |
| `BEGET_DEPLOY_METHOD` | не задано или `ftp` | Деплой по **FTP** (нужны секреты `BEGET_FTP_*`). |
| `BEGET_DEPLOY_METHOD` | `ssh` | Деплой через **`rsync` по SSH** (нужны секреты `BEGET_SSH_*`; секреты FTP не используются). |

## 3. События на галерее: planned / past и автоматическое обновление

Разделение на **«Что впереди»** и **«Прошедшие»** делается **при статической сборке** страницы `gallery.astro`: сравнивается дата события (`event_at`) с «сегодня» в заданной таймзоне (логика в `astro-app/src/lib/eventAt.ts`). Компонент **`EventsSlider`** только отображает переданные массивы — в браузере даты сами не пересчитываются.

Поэтому:

- после **пуша в `main`** сайт на Beget обновится из **CI**, и слайдер получит актуальные списки;
- после **полуночи** без нового коммита HTML не меняется, пока снова не выполнится **`npm run build`** — для этого и добавлен **ежедневный cron** в `rebuild-static.yml`.

Если нужно обновить слайдер в тот же день без ожидания ночного cron: запустите workflow **Rebuild static site** вручную или настройте webhook при публикации в Strapi.

## 4. Чеклист для нового окружения

1. [х] В Strapi создан токен для чтения, в GitHub добавлены `STRAPI_URL` и `STRAPI_TOKEN`.
2. [x] Выбран способ выкладки: **FTP** (секреты `BEGET_FTP_*`) или **SSH** (`BEGET_DEPLOY_METHOD=ssh` и секреты `BEGET_SSH_*`). Путь к корню сайта проверен: в нём после деплоя лежат `index.html`, `_astro`, папки страниц.
3. [x] Создана переменная `BEGET_DEPLOY=true` (или `BEGET_FTP_DEPLOY=true`), если нужен автодеплой.
4. [ ] Push в `main`: в Actions зелёный **CI**, при включённой выкладке — успешный **Deploy Astro to Beget**.
5. [ ] Раз в сутки по расписанию проходит **Rebuild static site** (проверьте логи после первой ночи).

**Замечание:** у GitHub по умолчанию отключаются **scheduled** workflow после длительной неактивности репозитория; при необходимости включите снова в интерфейсе Actions.

## 5. Сводка: что куда кладёт сайт

| Ситуация | Что происходит |
|----------|----------------|
| Открыт PR в `main` | CI гоняет проверки и сборки, **деплоя нет**, артефакт для Beget не создаётся. |
| Merge / push в `main` | CI успешен → при `BEGET_DEPLOY=true` или `BEGET_FTP_DEPLOY=true` → деплой на Beget (FTP или SSH по `BEGET_DEPLOY_METHOD`). |
| Правка в Strapi + webhook | **Rebuild static site** → сборка → при включённой выкладке → тот же метод деплоя. |
| Смена календарного дня без коммита | Ночной **schedule** в **Rebuild static site** → новая сборка с актуальными planned/past → выкладка при включённой переменной. |
| Без автовыкладки | Не задавайте `BEGET_DEPLOY` / `BEGET_FTP_DEPLOY` как `true`; скачивайте артефакт **astro-dist** и заливайте файлы вручную (см. [strapi-cloud-to-beget.md §6.1](./strapi-cloud-to-beget.md)). |
