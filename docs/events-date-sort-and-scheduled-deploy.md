# События: сортировка по дате и ежедневная пересборка (вариант A)

Вариант A — **машинная дата события** в Strapi как единственный источник для порядка карточек и для деления на «Что впереди» / «Прошедшие», плюс **регулярная сборка и деплой** статического сайта (иначе HTML не меняется сам по себе после полуночи).

Ниже — по шагам: CMS → фронт → автодеплой.

---

## 1. Идея

| Что нужно | Как достигается |
|-----------|-----------------|
| Сортировка в админке | Поле типа **дата/время** (`datetime`), по нему сортируют список в Strapi |
| Порядок в слайдере | Тот же порядок: запрос API с `sort=...` и/или `sort()` в Astro после маппинга |
| Автопереход «план → архив» | При **сборке** сравнить дату события с «сегодня» (в нужной таймзоне), не полагаться на ручное `planned`/`past` для логики слайдера |
| Актуальность каждый день | **Cron** в CI: раз в сутки `npm run build` и тот же деплой, что и при ручном rebuild |

Ручное поле `statusPlan` можно оставить для редких исключений или убрать из логики слайдера после внедрения даты — это отдельное решение продукта.

---

## 2. Strapi: поле даты и сортировка в админке

### 2.1. Новое поле в модели Event

В репозитории Strapi (CMS) схема лежит в `src/api/event/content-types/event/schema.json`. Добавьте атрибут, например:

- **`event_at`** — тип `datetime` (обязательное для новых событий; для старых записей позже заполните вручную или миграцией).

Оставьте **`date_label`** как строку для красивой подписи на карточке («15 марта 2026»), если она отличается от автоматического формата.

После изменения схемы:

1. Перезапустите Strapi локально / задеплойте бэкенд.
2. В панели Strapi выполните заполнение `event_at` для существующих событий.
3. При необходимости перегенерируйте типы: `npm run strapi ts:generate-types` (или как принято в вашем `package.json`).

### 2.2. Сортировка в списке админки

В **Strapi 5** откройте **Content Manager → Event**: в таблице записей можно включить сортировку по колонке (клик по заголовку), если поле отображается в списке.

Чтобы колонка появилась и сортировка была удобной:

- **Settings → Content-Type Builder → Event → Configure the view** (настройка отображения списка): добавьте **`event_at`** в видимые поля списка.

Дефолтную сортировку при открытии списка Strapi задаёт через UI сохранённое состояние пользователя; «жёсткий» дефолт на уровне только JSON-схемы ограничен — на практике достаточно API-сортировки при выборке (см. раздел 3) и ручной сортировки в таблице при редактировании.

### 2.3. Права API

Убедитесь, что `event_at` не скрыт от роли, которой пользуется Astro (токен `STRAPI_TOKEN` / публичные права на `find` для `events`), иначе поле не придёт в ответ.

## 3. Astro: запрос, разделение planned/past, порядок карточек

Файл с данными для слайдера: `astro-app/src/pages/gallery.astro` (сейчас: `sort=order:asc` и фильтр по `statusPlan`).

### 3.1. Запрос к Strapi

Используйте сортировку по дате, например:

```http
GET /api/events?populate=*&sort=event_at:asc
```

(точный префикс `/api` зависит от вашей обёртки `fetchStrapi` — главное — параметр `sort`.)

### 3.2. Логика «план / прошлое» при сборке

На этапе **SSG** (`astro build`) в скрипте страницы:

1. Парсите **`event_at`** в момент времени (ISO из Strapi).
2. Определите **границу «сегодня»** в одной таймзоне, например **`Asia/Yekaterinburg`** (Тюмень), чтобы «наступило число» совпадало с ожиданием редакции.
3. События с `event_at` **строго до** начала текущих суток в этой зоне → **past**; **с начала суток и позже** → **planned** (уточните правило: «в тот же календарный день ещё planned» или «до конца дня» — и зафиксируйте в коде одной функцией).

Библиотека вроде **`@date-fns/tz`** или нативный `Temporal` (если целевой Node поддерживает) упрощает сравнение; минимальный вариант — `Intl` + ручная нормализация, но его легче сломать на DST (для России обычно без DST).

### 3.3. Порядок внутри секций

Типичный UX:

- **Прошедшие** — от **новых к старым** (`event_at` по убыванию).
- **Что впереди** — от **ближайших к дальним** (`event_at` по возрастанию).

После `filter` вызовите `.sort()` по timestamp в каждом массиве.

Компонент **`EventsSlider`** менять не обязательно: он выводит массивы в том порядке, в котором их передали.

### 3.4. Страница события `events/[id].astro`

Сейчас «прошло / нет» для оформления берётся из `statusPlan`. После перехода на дату имеет смысл вычислять **`isPast`** так же, как на галерее (по `event_at` и той же таймзоне), чтобы не было рассинхрона между карточкой и страницей события.

## 4. Автодеплой: ежедневная пересборка

Статический сайт обновляется только когда снова выполнена **сборка** и **выкладка** артефакта. Вариант A предполагает **расписание** (cron).

В репозитории уже есть workflow **`.github/workflows/rebuild-static.yml`**:

- Собирает `astro-app` с секретами `STRAPI_URL`, `STRAPI_TOKEN`.
- При `vars.BEGET_FTP_DEPLOY == 'true'` выкладывает `dist` на Beget по FTP.

### 4.1. Добавить запуск по расписанию

В тот же файл в секцию `on:` добавьте **`schedule`** (время в **UTC**):

```yaml
on:
  repository_dispatch:
    types: [strapi_publish]
  workflow_dispatch:
  schedule:
    # Каждый день в 03:00 UTC ≈ 08:00 по Екатеринбургу (проверьте летнее/зимнее для вашего хоста; для РФ чаще постоянный сдвиг)
    - cron: '0 3 * * *'
```

Подберите `cron` так, чтобы пересборка проходила **после полуночи** в вашей таймзоне с запасом (например, ночью, когда мало трафика).

`concurrency` у вас уже настроен — параллельные запуски не должны копить очередь бессмысленно.

### 4.2. Секреты и переменные

Для успешного nightly job нужно то же, что и при ручном «Rebuild static site»:

| Назначение | Где |
|------------|-----|
| `STRAPI_URL`, `STRAPI_TOKEN` | **Settings → Secrets and variables → Actions → Secrets** |
| Выкладка на Beget | Переменная `BEGET_FTP_DEPLOY = true` и секреты `BEGET_FTP_*` — как в `docs/strapi-cloud-to-beget.md` |

Без деплоя артефакт только соберётся и сохранится как **artifact** в GitHub Actions — сайт на хостинге не обновится.

### 4.3. Альтернативы без ежедневного cron

- **Webhook от Strapi** при публикации — у вас уже есть `repository_dispatch` с типом `strapi_publish`: контент обновляется сразу после правок, но **смена дня без правок в CMS** не триггерит сборку — для этого как раз нужен **schedule**.
- Только **ручной** workflow_dispatch — даты на сайте устаревают, пока кто-то не нажмёт rebuild.

## 6. Ограничения варианта A

- Задержка до **следующего запуска cron** (если cron раз в сутки — максимум ~24 ч «ошибки» относительно мгновенной смены в полночь; обычно приемлемо).
- Нужен **живой Strapi** на момент сборки CI; если CMS недоступна, сборка упадёт или отдаст кэш — зависит от вашего `fetchStrapi`.

Если понадобится обновление **в тот же день без ожидания cron**, оставьте возможность вручную запускать **Rebuild static site** или настройте webhook при публикации (у вас уже частично есть).
